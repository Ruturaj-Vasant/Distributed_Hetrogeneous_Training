# PROJECT OVERVIEW

Building a distributed heterogeneous training system for ResNet-101 on Tiny ImageNet-200.
Multiple laptops (Mac MPS + Windows CUDA) connected via Tailscale VPN.
Architecture: Parameter Server (NOT PyTorch DDP). Leader holds model, workers compute
gradients, push compressed gradients to leader, leader aggregates + steps optimizer,
returns weight delta to workers.

Why not DDP: MPS and CUDA cannot share NCCL/Gloo backends. DDP requires homogeneous
hardware and serializes on slowest worker. Parameter server handles heterogeneous devices
and unequal batch sizes naturally.

# TECHNICAL ARCHITECTURE

## Communication
- Transport: gRPC over Tailscale WireGuard VPN
- Leader port: 50051 (default)
- Protocol: protobuf (proto/trainer.proto)
- Message size limit: 256 MB on both server (leader.py) and client channels (worker.py)
- Leader disables grpc.so_reuseport to prevent multiple leader processes from sharing 50051
- Heartbeat interval: 5 seconds (asyncio.sleep(5) in worker heartbeat_loop)
- Heartbeat timeout: ~15 seconds before worker marked DEAD on leader

## gRPC Channel Design (worker)
- async_ch = aio.insecure_channel(...) → for Heartbeat (bidi stream) + GetAssignment (blocking unary)
- sync_ch = grpc.insecure_channel(...) → for ExchangeGradients (called from training thread)
- Reason: training loop runs in asyncio.get_event_loop().run_in_executor(), can't use aio there
- Startup connection path now waits for async_ch.channel_ready(), retries registration
  with fresh channels, and supports CLI tuning:
  --connect-timeout, --rpc-timeout, --connect-retries, --retry-backoff,
  --max-retry-backoff

## Leader internals
- asyncio.Lock (_lock): guards _workers dict
- asyncio.Condition (_grad_cond): gradient sync — last worker to arrive aggregates + notifies all
- asyncio.Event (_training_started): set when operator types 'start' or auto-start fires
- Per-worker asyncio.Event (assignment_event): set by start_training(), unblocks GetAssignment
- heartbeat_monitor(): background task, removes timed-out workers, handles dead-worker mid-sync

## Data flow per training step
1. Worker: forward pass → loss.backward() → compress_gradients() [Top-K per layer]
2. Worker: ExchangeGradients RPC → sends GradientPush (worker_id, step, loss, sparse_grads)
3. Leader: collects from all alive workers (asyncio.Condition wait loop)
4. Leader: last worker triggers _aggregate_and_step() → weighted average → optimizer.step()
5. Leader: serializes weight delta (new_weights - old_weights) → sends WeightUpdate
6. Worker: apply_delta() → adds delta to model params in-place

## Gradient compression
Top-K sparsification: for each layer with gradient, keep topk_k elements by absolute value.
Sends (indices, values, shape) tuples. Default topk_k=50000.
Full gradient (topk_k=0) also supported.
Gradients moved to CPU + cast float32 before sending (avoids MPS/CUDA serialization issues).

## Sharding
worker_fraction = worker_score / sum(all_scores)
worker_samples = int(fraction × 100,000)  [Tiny ImageNet train size]
batch_size scales proportionally (base 32, scaled by fraction, min 8, max 256)

## Model
ResNet-101 default (also supports resnet18, resnet50 via --model flag)
Modified final FC: 200 classes (Tiny ImageNet-200)
Initial weights serialized with torch.save() → sent as bytes in GetAssignmentResponse
First sync: full state dict. Subsequent: weight delta only.

## Tailscale

Worker DNS: devanshis-macbook-air.taila5426e.ts.net
Both machines must be logged into same Tailscale account


# KEY DECISIONS & CONSTRAINTS

1. Parameter Server over DDP: MPS + CUDA can't use NCCL. DDP serializes on slowest.
   PS allows heterogeneous hardware, unequal batch sizes, natural fault tolerance.

2. Two gRPC channels per worker (async + sync): asyncio can't use sync grpc stubs.
   Training thread uses run_in_executor → needs sync channel for ExchangeGradients.

3. Weight delta (not full state dict) per step: reduces bandwidth from ~170 MB to ~1-10 MB.
   Full state dict only on initial assignment.

4. asyncio.Condition for gradient sync: "last worker aggregates" semantics.
   Lock ordering: _grad_cond → _lock (never reversed to avoid deadlock).

5. grpc.aio has NO sleep() function in grpc >= 1.32+ (removed from public API).
   Must use asyncio.sleep() in async contexts, not aio.sleep().

6. Leader gRPC server needs explicit message size options:
   aio.server(options=[("grpc.max_receive_message_length", 256*1024*1024), ...])
   Default is 4 MB which rejects ResNet gradient pushes (~6 MB per step).

7. Non-interactive leader mode: when stdin is not a TTY (subprocess/CI), use
   server.wait_for_termination() instead of cli_loop() to avoid immediate exit.

8. Dataset is pre-downloaded on each worker independently (no shared filesystem).
   Workers download to ~/.cache/tiny-imagenet-200. Script skips if already present.

9. MPS device: Apple Silicon GPU. Cannot use NCCL. Supported via torch.device("mps").
   Fallback chain: CUDA → MPS → CPU.

10. gRPC SETTINGS timeout diagnosis: when nc succeeds but grpc channel_ready() times out,
    check for duplicate/stopped leader processes with lsof/ps. On 2026-05-08 the cause
    was two leaders bound to 50051 due to grpc port reuse; one was STAT T. Fixed in
    leader.py via ("grpc.so_reuseport", 0). Tailscale ping and TCP were fine.

# KNOWN ISSUES / RISKS

1. [FIXED IN CODE / VERIFY IN CLEAN RUN] gRPC SETTINGS timeout on Tailscale path.
   Symptom: nc succeeds, grpc.aio.Register() or channel_ready() times out.
   Confirmed cause on 2026-05-08: duplicate leader.py processes sharing 50051; one
   process was stopped (STAT T). Workers could connect to the stopped listener and
   hang before HTTP/2 SETTINGS.
   Fixes: leader disables grpc.so_reuseport; worker waits for channel readiness,
   retries registration, and logs clean connection failures.
   Remaining operational step: kill currently stale listeners before starting patched leader.

2. [FIXED] grpc.aio has no sleep() — heartbeat crashed silently after first beat.
   Workers showed as DEAD ~15s after connecting because hb_task raised AttributeError.
   Fixed: asyncio.sleep(5) in heartbeat_loop. Exception handler broadened to catch all.
   Commit: 83dd643

3. [FIXED] gRPC 4 MB message limit on leader server.
   ResNet gradient push ~6 MB triggered RESOURCE_EXHAUSTED.
   Fixed: aio.server(options=[("grpc.max_receive_message_length", 256*1024*1024)]).
   Commit: 089213e

4. [FIXED] Leader exited immediately in subprocess mode.
   cli_loop() called input() which raised EOFError when stdin=DEVNULL.
   Fixed: sys.stdin.isatty() check → use server.wait_for_termination() for non-TTY.

5. [RISK] ResNet-101 full weight delta still ~44 MB per step (44,131 KB observed in smoke test).
   For many workers or slow links this could bottleneck gradient exchange.
   Mitigation: Top-K compression reduces gradient payload, but weight delta is still full model.
   Future: consider sending only changed layers or using lower precision (fp16).

6. [MITIGATED] Dead worker mid-sync: heartbeat_monitor handles forced aggregation
   for both dead-worker and grad_sync_timeout cases. If ALL workers die and no
   gradients were received, the round is not forced (no-op). Reasonable tradeoff.

7. [RISK] No gradient synchronization across epochs — each step is independent.
   If a worker falls behind, it gets stale weight deltas. Acceptable for now.

8. [KNOWN LIMITATION] Windows bootstrap script not yet tested end-to-end on real hardware.
   CUDA detection via nvidia-smi heuristic may need tuning for specific driver versions.

9. [KNOWN LIMITATION] MPS + asyncio.run_in_executor: MPS operations in a thread pool
   executor may have restrictions. If MPS training hangs, may need to pin to main thread.
